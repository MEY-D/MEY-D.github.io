Now we have two windows ‚Äî they are same-origin and have a parent/child relationship, so they can read from each other.
Based on the location.href in the window, XSS payload decides to either history.go(-2) or read the data.

One last trick is using a query parameter in the URL. That‚Äôs because the disk cache is based on the full URL ‚Äî so by using different query parameters, we can keep /profile?test cached even when we later redirect the victim to the same endpoint /profile For example:

When It Doesn’t Work

One of my friends Moein solved the challenge and asked me: “When does this not work?”
And he gave me this:

After very little testing, I found that only:

Cache-Control: no-store

is enough to prevent such an attack ‚Äî as explained in:
üîó https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Cache-Control
